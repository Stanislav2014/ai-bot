# Открытые вопросы / решения

Архитектурные и продуктовые вопросы без явного ответа. Каждый пункт — проблема + варианты + (если есть) рекомендация.

## 1. Persistence для per-user selected model

**Проблема**: `BotHandlers.user_models` — `dict` в памяти процесса. При рестарте все выбранные пользователями модели сбрасываются на дефолт. Это «по спеке» (stateless), но UX не очень.

**Варианты**:
- A. Оставить как есть — осознанный stateless
- B. Добавить лёгкий JSON файл на диске (`data/user_models.json`)
- C. Добавить SQLite (одна таблица `user_preferences`)

**Рекомендация**: B если реально беспокоит; сейчас — A.

---

## 2. Dialog history

**Проблема**: Каждое сообщение — независимый LLM-запрос. Пользователь не может задать follow-up вопрос. Это осознанный stateless в MVP, но частая жалоба.

**Варианты**:
- A. Никогда не добавлять — остаётся stateless
- B. In-memory sliding window (последние N сообщений per user, теряется при рестарте)
- C. Persistent history (SQLite)

**Рекомендация**: не торопиться. Следующий шаг — собрать реальный usage feedback.

---

## 3. Ollama vs Lemonade — финальный выбор

**Проблема**: docker-compose поднимает `lemonade`, но `app/config.py` имеет default `http://ollama:11434`, и README описывает Ollama. Миграция не завершена. См. [legacy-warning.md](legacy-warning.md) § 1.

**Варианты**:
- A. Финализировать миграцию на Lemonade — обновить config, README, Makefile
- B. Откатиться на Ollama
- C. Поддерживать оба — параметризовать LLM_BASE_URL (уже частично сделано)

**Рекомендация**: A. Lemonade более унифицирован, уже в docker-compose.

## 4. Auth / access control

**Проблема**: Любой, кто знает username бота, может им пользоваться. Токены LLM-сервера расходуются без ограничений.

**Варианты**:
- A. Allowlist Telegram user_id в .env
- B. Invite-token при `/start`
- C. Rate-limit per user (без allowlist)
- D. Ничего — бот внутри доверенной среды

**Рекомендация**: A если развёртывается публично; сейчас — D.

---

## 5. Long LLM requests > polling timeout

**Проблема**: `llm_timeout=120s`, но Telegram polling updater может таймаутить раньше. Если LLM отвечает >60s, может быть рассинхрон.

**Варианты**:
- A. Снизить `llm_timeout` до 60s
- B. Отправлять промежуточные typing actions каждые N секунд (сейчас typing отправляется один раз)
- C. Оставить как есть — на практике проблем не видно

**Рекомендация**: B + понаблюдать.

---

## 6. Model list caching

**Проблема**: `list_models()` вызывается на `/models` и при `/model <name>` / callback, каждый раз делает HTTP запрос к LLM серверу. При высокой нагрузке это зря.

**Варианты**:
- A. Кеш на 60s (TTL)
- B. Кеш на всё время жизни процесса с invalidation по сигналу
- C. Оставить — пока нагрузки нет

**Рекомендация**: A, очень дешёвое улучшение.
