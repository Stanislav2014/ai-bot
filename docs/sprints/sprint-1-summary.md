# Итог Sprint 1 — человеческим языком

Короткий пересказ без жаргона. Полный delivery-документ с кодом и примерами — [sprint-1-delivery.md](sprint-1-delivery.md).

## Как хранится история

Каждому пользователю — свой YAML-файл `data/history/{user_id}.yaml` со списком сообщений. При каждом новом сообщении бот загружает этот файл, шлёт всё в модель вместе с новым вопросом, а ответ дописывает обратно. Старые сообщения автоматически срезаются, если превышен лимит (по числу или по символам). `/reset` удаляет файл. Флаг `HISTORY_ENABLED=false` делает бота полностью беспамятным.

## Как работает суммаризация

Когда история становится длиннее порога (например, 20 сообщений), бот делает отдельный запрос к LLM: «сожми этот диалог в 1-2 предложения». Старые сообщения заменяются одной строкой-резюме, последние N сообщений остаются как есть. При следующем триггере старое резюме сливается с новыми сообщениями в новое — цепочки не накапливаются. Если LLM упал — возвращаем оригинал, юзер ничего не замечает.

## Какие проблемы возникли

1. Коммиты двух разных задач перемешались — разобрал, ветку пересобрал.
2. Я чистил `change-request.md` после merge — оказалось, он должен зеркалить весь спринт.
3. Советовал «поставь все лимиты в 0 → stateless» — это дало ровно наоборот: безлимитную память. Пришлось делать отдельный флаг `HISTORY_ENABLED`.
4. Лог-файл писался в контейнер под root — сделал bind-mount в проект.
5. System prompt был захардкожен по-английски — вынес в env.
6. В логе не было видно, что реально уходит в модель — добавил полный payload.
